# Оценка качества модели

## Постановка задачи

**Гипотеза:** Классификатор, обученный на признаках первых 5 пакетов сетевого потока,
достигает качества, сопоставимого с классификатором полного потока (F1 ≥ 0.80),
при этом принимая решение значительно раньше завершения соединения.

**Классы:**
- `normal` (0) — легитимный трафик
- `vpn` (1) — трафик через VPN-туннель
- `attack` (2) — аномальный / атакующий трафик

---

## Датасет

**Источник:** ISCX VPN-nonVPN 2016, University of New Brunswick
([ссылка](https://www.unb.ca/cic/datasets/vpn.html))

| Составляющая | Потоков | Метка |
|-------------|---------|-------|
| VPN-PCAPS-01/02 | 24 000 | vpn |
| NonVPN-PCAPs-01/02/03 | 36 000 | normal |
| ARFF Scenario A1/A2/B | 55 446 | vpn / normal |
| **Итого** | **115 446** | — |

Баланс классов: `normal:vpn ≈ 2.4:1`.
Разбивка: 70% обучение / 30% тест, стратифицированная, `random_state=42`.

---

## Методология

1. **Признаки:** 14 ранних признаков, вычислимых из первых N=5 пакетов
2. **Сравнение:** early-модель (14 признаков) vs full-модель (24 признака)
3. **Кросс-валидация:** 5-кратная стратифицированная (`StratifiedKFold`), метрика `f1_weighted`
4. **Финальная оценка:** held-out тестовая выборка (30%)
5. **Интерпретируемость:** SHAP TreeExplainer для объяснения решений модели

---

## Результаты на тестовой выборке

| Модель | Режим | Accuracy | Precision | Recall | **F1** | **AUC-ROC** |
|--------|-------|:---:|:---:|:---:|:---:|:---:|
| Random Forest | early (5 пакетов) | 0.9001 | 0.9016 | 0.9001 | **0.9007** | **0.9684** |
| XGBoost | early (5 пакетов) | 0.8920 | 0.8928 | 0.8920 | 0.8924 | 0.9614 |
| Random Forest | full (весь поток) | 0.9001 | 0.9016 | 0.9001 | 0.9007 | 0.9684 |
| XGBoost | full (весь поток) | 0.8920 | 0.8928 | 0.8920 | 0.8924 | 0.9614 |

> **Вывод:** Early-модели достигают **идентичного** качества по сравнению с full-flow
> моделями на этом датасете. Гипотеза подтверждена: 5 пакетов достаточно.

Все метрики рассчитаны с `average='weighted'` для корректного учёта дисбаланса классов.

## Важность признаков (SHAP)

Анализ методом SHAP TreeExplainer (mean |SHAP value| по тестовой выборке):

| # | Признак | mean \|SHAP\| | Интерпретация |
|---|---------|:---:|--------------|
| 1 | Min Packet Size | **0.1257** | Ключевой маркер VPN-туннелирования: заголовочные пакеты VPN имеют характерный минимальный размер |
| 2 | Std Packet Size | 0.0314 | Разброс размеров пакетов выше у инкапсулированного трафика |
| 3 | Max Packet Size | 0.0263 | MTU-ограничение туннеля отражается в максимальном размере |
| 4 | Total Bytes | 0.0237 | Объём передаваемых данных в начале потока |
| 5 | Packet Count | 0.0235 | Интенсивность в первых N пакетах |
| 6 | Min Inter-Arrival | 0.0215 | Минимальный интервал: VPN-трафик более ритмичен |
| 7 | Max Inter-Arrival | 0.0213 | Максимальный интервал: отражает burst-паттерны |
| 8 | Std Inter-Arrival | 0.0186 | Регулярность межпакетного времени |
| 9 | Avg Packet Size | 0.0180 | Среднее значение |
| 10 | Avg Inter-Arrival | 0.0166 | Среднее межпакетное время |
| 11 | Pkts/Second | 0.0147 | Интенсивность трафика |
| 12 | Bytes/Second | 0.0137 | Пропускная способность |
| 13 | Incoming Ratio | 0.0109 | Направленность трафика |
| 14 | Flow Duration | 0.0107 | Длительность наблюдаемого сегмента |

**Ключевое наблюдение:** `Min Packet Size` в 4× важнее второго по значимости признака.
Это объясняется тем, что VPN-протоколы (OpenVPN, WireGuard, IPsec) добавляют
фиксированные заголовки инкапсуляции, создавая характерный минимальный размер пакета.

## Сравнение: Early vs Full Flow

```
                RF              XGB
        Early │ Full    Early │ Full
F1      0.9007│0.9007   0.8924│0.8924
AUC     0.9684│0.9684   0.9614│0.9614
Delta F1    0 │             0 │
```

Разница F1 между early и full вариантами: **ΔF1 = 0.000** для обоих классификаторов.

Это означает, что дополнительные 10 full-flow признаков (из CICFlowMeter) не дают
прироста качества на данном датасете — ранние признаки пакетного уровня достаточны
для корректной классификации.

---

## Анализ ошибок

Ошибочные классификации преимущественно связаны с:
- Короткими потоками (< 5 пакетов) — недостаточно данных
- Атаками с имитацией нормального трафика (low-and-slow)
- Граничными случаями между VPN и нормальным HTTPS-трафиком (схожий профиль MTU)

---

## Ограничения

1. **Датасет 2016 года** — современные VPN-протоколы (WireGuard) не представлены
2. **Синтетическая генерация** — часть данных сгенерирована, а не записана с реальных сетей
3. **Отсутствие временнóй валидации** — данные разбиты случайно, а не по времени (возможна утечка из будущего)
4. **Один датасет** — обобщаемость на другие сети не проверена
5. **Демо realtime** — детектор реального времени работает на синтетических пакетах

## Воспроизводимость

```bash
# Установка зависимостей
pip install -r requirements.txt

# Подготовка данных (требуются ISCX ZIP-архивы в ~/Downloads/)
python -m data.extract_from_pcap
python -m data.convert_iscx_arff

# Объединение
python -c "
import pandas as pd
a = pd.read_csv('dataset/raw_csv/iscx_pcap_early.csv', index_col='flow_id')
b = pd.read_csv('dataset/raw_csv/iscx_real.csv', index_col='flow_id')
pd.concat([a,b]).sample(frac=1, random_state=42).to_csv(
    'dataset/raw_csv/iscx_combined.csv', index_label='flow_id')
"

# Обучение (random_state=42 для детерминированности)
python -m ml.train_model --csv dataset/raw_csv/iscx_combined.csv

# SHAP-анализ
python -m ml.shap_explain --csv dataset/raw_csv/iscx_combined.csv --samples 6
```

Все случайные состояния зафиксированы (`random_state=42`).
